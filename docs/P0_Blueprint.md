# OnIt P0 Blueprint

**版本**：v1.0  
**日期**：2026-04-05  
**状态**：首轮交付基线

---

## 1. 产品一句话定义

OnIt 是一个 **AI 团队协调层**：用户贴入材料、brief 或目标，系统先理解输入，再抽取岗位职能与任务结构，随后按具体任务匹配或雇佣合适的 Agent，之后进入执行、阻塞、恢复与结果回流。

> 用户只负责表达 **What** 与 **Quality**，不负责系统内部的 **How** 与 **Who**。

---

## 2. 对象模型

OnIt 的核心对象体系由以下四层构成，每层有明确的职责边界：

| 对象 | 中文名 | 职责 | 不是什么 |
|---|---|---|---|
| `Brief` | 用户输入 | 用户贴入的材料、目标或 brief，是整条主链的起点 | 不是表单、不是配置项 |
| `Mission` | 任务 | 系统从 Brief 中抽取的结构化任务单元，包含目标、职能需求与验收标准 | 不是旧 workspace 的 project |
| `AgentRole` | Agent 角色 | 被匹配到某个 Mission 的执行者，有明确的职能描述、状态与交付物 | 不是旧 Agent 卡片 |
| `Execution` | 执行记录 | AgentRole 的执行过程，包含状态流转、阻塞点、恢复记录与输出 | 不是 Paperclip 的 issue 直接映射 |

### 对象关系图

```
Brief (用户输入)
  └─ Mission (任务结构，由总管从 Brief 中抽取)
       └─ AgentRole[] (每个任务匹配的执行者)
            └─ Execution (执行状态、阻塞、交付物)
```

**总管（Manager）** 是协调层的唯一入口，负责：
1. 理解 Brief，抽取 Mission 结构
2. 为每个 Mission 匹配 AgentRole
3. 管理 Execution 生命周期（pending → running → blocked → done/failed）
4. 在 HitL 节点暂停并等待用户决策
5. 汇总结果并回流给用户

---

## 3. 默认入口

**唯一合法的默认入口**：用户输入 Brief（材料、目标或任务描述）。

```
[用户] → 输入 Brief → [总管理解] → [Mission 抽取] → [Agent 匹配] → [执行] → [结果回流]
```

**禁止的入口形式**：
- 旧 workspace 首页（项目列表 + 新建按钮）
- Agent 面板（先展示 Agent 卡片让用户选择）
- 流程图编辑器（让用户手工搭流程）
- 表单填写（让用户填写复杂结构化信息）

## 4. 主流程（Golden Case 主链）

以 Golden Case「AI 购物 App 创作者招募」为基准，主链如下：

```
Step 1: 用户输入 Brief
  → 用户贴入目标描述（可以是一段话、一份文档或一个链接）

Step 2: 总管理解 Brief
  → 系统解析输入，识别核心目标、约束条件、所需能力
  → 输出结构化理解摘要（展示给用户确认）

Step 3: Mission 抽取与 Agent 匹配
  → 总管将目标拆解为 N 个 Mission
  → 为每个 Mission 匹配最合适的 AgentRole（按职能、能力标签）
  → 展示团队方案，等待用户确认（HitL Gate 1）

Step 4: 执行推进
  → 各 AgentRole 按顺序或并行执行
  → 总管持续汇报进度（running → partial done → ...）
  → 遇到需要用户决策的节点时暂停（blocked）

Step 5: HitL 决策节点
  → 总管展示选项或需要用户审批的内容
  → 用户做出选择后，执行恢复（recovery）

Step 6: 结果回流
  → 所有 Mission 完成后，总管汇总结果
  → 展示交付物列表与执行摘要
```

## 6. 状态流转

### Mission 状态机

```
IDLE → UNDERSTANDING → PROPOSING → CONFIRMING → EXECUTING → COMPLETE
                                       ↕
                                    BLOCKED → (用户决策) → EXECUTING
```

| 状态 | 含义 | 触发条件 |
|---|---|---|
| `IDLE` | 初始状态 | Brief 刚提交 |
| `UNDERSTANDING` | 总管正在理解 Brief | 系统处理中 |
| `PROPOSING` | 总管正在制定方案 | 理解完成，生成团队方案 |
| `CONFIRMING` | 等待用户确认方案 | 方案生成完毕，HitL Gate 1 |
| `EXECUTING` | 执行中 | 用户确认后 |
| `BLOCKED` | 等待用户决策 | 遇到 HitL 节点 |
| `COMPLETE` | 全部完成 | 所有 AgentRole 完成 |

### AgentRole 状态机

```
pending → running → done
              ↓
           blocked → (用户决策) → running
              ↓
           failed
```

## 7. Human-in-the-Loop 设计原则

HitL 是 OnIt 主流程中的**正式暂停点**，不是可选附加能力。

| HitL 类型 | 触发场景 | 用户操作 |
|---|---|---|
| **Gate 1：方案确认** | 总管提出团队方案后 | 确认 / 修改 / 拒绝 |
| **Gate 2：内容审批** | Agent 产出需要用户审核的内容 | 选择版本 / 提供反馈 |
| **Gate 3：关键决策** | 执行中遇到需要用户判断的分叉 | 选择方向 |
| **Gate 4：结果确认** | 全部完成后 | 确认接收 / 要求修改 |

## 8. 设计语言基线

| 维度 | 决策 |
|---|---|
| **主题** | 深色背景（`#0A0A0A`），低噪声，高对比 |
| **字体** | Inter，系统感强，无衬线 |
| **主色** | `oklch(0.65 0.2 250)`（蓝紫色，科技感） |
| **成功色** | `oklch(0.65 0.15 145)`（绿色） |
| **警告色** | `oklch(0.75 0.15 60)`（琥珀色） |
| **圆角** | `12px`（卡片），`20px`（气泡） |
| **动效** | Framer Motion，入场 `y: 8 → 0`，`duration: 0.25s` |
| **布局** | 左右分栏（对话 + 状态），移动端折叠为单列 |
